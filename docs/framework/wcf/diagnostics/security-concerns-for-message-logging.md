### <a name="logging-sensitive-information"></a><span data-ttu-id="04e4a-105">Hassas bilgileri günlüğe kaydetme</span><span class="sxs-lookup"><span data-stu-id="04e4a-105">Logging Sensitive Information</span></span>  
 [!INCLUDE[indigo1](../../../../includes/indigo1-md.md)]<span data-ttu-id="04e4a-106">uygulamaya özgü üstbilgi ve gövde herhangi bir veri değiştirmez.</span><span class="sxs-lookup"><span data-stu-id="04e4a-106"> does not modify any data in application-specific headers and body.</span></span> [!INCLUDE[indigo2](../../../../includes/indigo2-md.md)]<span data-ttu-id="04e4a-107">Ayrıca uygulamaya özgü üstbilgileri veya gövde verileri kişisel bilgilerin izlemez.</span><span class="sxs-lookup"><span data-stu-id="04e4a-107"> also does not track personal information in application-specific headers or body data.</span></span>  
  
 <span data-ttu-id="04e4a-108">İleti günlüğe kaydetme etkinleştirildiğinde, uygulamaya özgü üstbilgiler, sorgu dize gibi kişisel bilgilerinizi; ve kredi kartı numarası gibi bilgileri gövde, günlüklerde görünür hale gelebilir.</span><span class="sxs-lookup"><span data-stu-id="04e4a-108">When message logging is enabled, personal information in application-specific headers, such as, a query string; and body information, such as, a credit card number, can become visible in the logs.</span></span> <span data-ttu-id="04e4a-109">Erişim denetimini yapılandırma ve günlük dosyaları uygulama için uygulama dağıtıcı sorumludur.</span><span class="sxs-lookup"><span data-stu-id="04e4a-109">The application deployer is responsible for enforcing access control on the configuration and log files.</span></span> <span data-ttu-id="04e4a-110">Bu tür bilgilerin görünür olmasını istemiyorsanız günlüğü devre dışı bırakın veya günlükleri paylaşmak istiyorsanız verilerin bir kısmını filtre gerekir.</span><span class="sxs-lookup"><span data-stu-id="04e4a-110">If you do not want this kind of information to be visible, you should disable logging, or filter out part of the data if you want to share the logs.</span></span>  
  
 <span data-ttu-id="04e4a-111">Aşağıdaki ipuçları bir günlük dosyasının içeriğinin istenmeden açıklanmasını engellemek yardımcı olabilir:</span><span class="sxs-lookup"><span data-stu-id="04e4a-111">The following tips can help you to prevent the content of a log file from being exposed unintentionally:</span></span>  
  
-   <span data-ttu-id="04e4a-112">Tarafından erişim denetim listeleri (ACL) hem de Web ana bilgisayarı ve kendi kendini barındıran senaryoları korunan dosyaları günlük emin olun.</span><span class="sxs-lookup"><span data-stu-id="04e4a-112">Ensure that the log files are protected by Access Control Lists (ACL) both in Web-host and self-host scenarios.</span></span>  
  
-   <span data-ttu-id="04e4a-113">Web isteği kullanarak kolayca sunulamıyor bir dosya uzantısı seçin.</span><span class="sxs-lookup"><span data-stu-id="04e4a-113">Choose a file extension that cannot be easily served using a Web request.</span></span> <span data-ttu-id="04e4a-114">Örneğin, .xml dosya uzantısı güvenli bir seçenek değil.</span><span class="sxs-lookup"><span data-stu-id="04e4a-114">For example, the .xml file extension is not a safe choice.</span></span> <span data-ttu-id="04e4a-115">Sunulabilecek uzantıların listesini görmek için Internet Information Services (IIS) yönetim kılavuzuna bakabilirsiniz.</span><span class="sxs-lookup"><span data-stu-id="04e4a-115">You can check the Internet Information Services (IIS) administration guide to see a list of extensions that can be served.</span></span>  
  
-   <span data-ttu-id="04e4a-116">Bir Web tarayıcısı kullanılarak harici bir tarafça erişilmesini önlemek üzere Web ana bilgisayar vroot genel dizininin dışında olması gereken günlük dosyası konumu için mutlak bir yol belirtin.</span><span class="sxs-lookup"><span data-stu-id="04e4a-116">Specify an absolute path for the log file location, which should be outside of the Web host vroot public directory to prevent it from being accessed by an external party using a Web browser.</span></span>  
  
 <span data-ttu-id="04e4a-117">Varsayılan olarak, anahtarları ve kullanıcı adı ve parola gibi kişisel bilgileri (PII) içindeki oturum açmadığı ve iletileri günlüğe.</span><span class="sxs-lookup"><span data-stu-id="04e4a-117">By default, keys and personally identifiable information (PII) such as username and password are not logged in traces and logged messages.</span></span> <span data-ttu-id="04e4a-118">Bir Makine Yöneticisi, ancak kullanabilirsiniz `enableLoggingKnownPII` özniteliğini `machineSettings` makinede çalışan uygulamaların bilinen kişisel bilgileri (PII) oturum izin vermek için Machine.config dosyasının öğesi.</span><span class="sxs-lookup"><span data-stu-id="04e4a-118">A machine administrator, however, can use the `enableLoggingKnownPII` attribute in the `machineSettings` element of the Machine.config file to permit applications running on the machine to log known personally identifiable information (PII).</span></span> <span data-ttu-id="04e4a-119">Aşağıdaki yapılandırma bunun nasıl yapılacağı gösterilmektedir:</span><span class="sxs-lookup"><span data-stu-id="04e4a-119">The following configuration demonstrates how to do this:</span></span>  
  
```xml  
<configuration>  
   <system.serviceModel>  
      <machineSettings enableLoggingKnownPii="true"/>  
   </system.serviceModel>  
</configuration>   
```  
  
 <span data-ttu-id="04e4a-120">Bir uygulama dağıtıcı sonra kullanabileceğiniz `logKnownPii` özniteliği PII gibi günlüğe kaydetmeyi etkinleştirmek için App.config veya Web.config dosyasında:</span><span class="sxs-lookup"><span data-stu-id="04e4a-120">An application deployer can then use the `logKnownPii` attribute in either the App.config or Web.config file to enable PII logging as follows:</span></span>  
  
```xml  
<system.diagnostics>  
  <sources>  
      <source name="System.ServiceModel.MessageLogging"  
        logKnownPii="true">  
        <listeners>  
                 <add name="messages"  
                 type="System.Diagnostics.XmlWriterTraceListener"  
                 initializeData="c:\logs\messages.svclog" />  
          </listeners>  
      </source>  
    </sources>  
</system.diagnostics>  
```  
  
 <span data-ttu-id="04e4a-121">Yalnızca her iki ayarın olduğunda `true` PII günlük kaydı etkin.</span><span class="sxs-lookup"><span data-stu-id="04e4a-121">Only when both settings are `true` is PII logging enabled.</span></span> <span data-ttu-id="04e4a-122">İki anahtar birleşimi her uygulama için bilinen PII oturum esnekliği sağlar.</span><span class="sxs-lookup"><span data-stu-id="04e4a-122">The combination of two switches allows the flexibility to log known PII for each application.</span></span>  
  
> [!IMPORTANT]
>  <span data-ttu-id="04e4a-123">İçinde [!INCLUDE[netfx_current_long](../../../../includes/netfx-current-long-md.md)] `logEntireMessage` ve `logKnownPii` bayrakları da ayarlanmalıdır `true` olarak Web.config dosyasını veya PII günlük kaydını etkinleştirmek için App.config dosyasında, aşağıdaki örnekte Göster `<system.serviceModel><messageLogging logEntireMessage="true" logKnownPii="true" …`.</span><span class="sxs-lookup"><span data-stu-id="04e4a-123">In [!INCLUDE[netfx_current_long](../../../../includes/netfx-current-long-md.md)] the `logEntireMessage` and `logKnownPii` flags must also be set to `true` in the Web.config file or the App.config file to enable PII logging, as show in the following example `<system.serviceModel><messageLogging logEntireMessage="true" logKnownPii="true" …`.</span></span>  
  
 <span data-ttu-id="04e4a-124">İki veya daha fazla özel kaynağı bir yapılandırma dosyası belirtirseniz, yalnızca ilk kaynak özniteliklerini okuma bilmeniz gerekir.</span><span class="sxs-lookup"><span data-stu-id="04e4a-124">You should be aware that if you specify two or more custom sources in a configuration file, only the attributes of the first source are read.</span></span> <span data-ttu-id="04e4a-125">Diğerleri yoksayılır.</span><span class="sxs-lookup"><span data-stu-id="04e4a-125">The others are ignored.</span></span> <span data-ttu-id="04e4a-126">Bu, PII günlüğü açıkça ikinci kaynağı için etkin olsa bile aşağıdaki App.config için dosya, PII hem kaynakları için günlüğe kaydedilmez, anlamına gelir.</span><span class="sxs-lookup"><span data-stu-id="04e4a-126">This means that, for the following App.config, file, PII is not logged for both sources even though PII logging is explicitly enabled for the second source.</span></span>  
  
```xml  
<system.diagnostics>  
   <sources>  
      <source name="System.ServiceModel.MessageLogging"  
              logKnownPii="false">  
              <listeners>  
                 <add name="messages"  
                      type="System.Diagnostics.XmlWriterTraceListener"  
                      initializeData="c:\logs\messages.svclog" />  
              </listeners>  
            </source>  
      <source name="System.ServiceModel"   
              logKnownPii="true">  
              <listeners>  
                 <add name="traces"  
                      type="System.Diagnostics.XmlWriterTraceListener"  
                      initializeData="c:\logs\traces.svclog" />  
              </listeners>  
      </source>  
   </sources>  
</system.diagnostics>  
```  
  
 <span data-ttu-id="04e4a-127">Varsa `<machineSettings enableLoggingKnownPii="Boolean"/>` öğesi dışında Machine.config dosyasının var, sistem oluşturur bir <xref:System.Configuration.ConfigurationErrorsException>.</span><span class="sxs-lookup"><span data-stu-id="04e4a-127">If the `<machineSettings enableLoggingKnownPii="Boolean"/>` element exists outside the Machine.config file, the system throws a <xref:System.Configuration.ConfigurationErrorsException>.</span></span>  
  
 <span data-ttu-id="04e4a-128">Uygulama başlatıldığında veya yeniden başlatır değişiklikler etkili olur.</span><span class="sxs-lookup"><span data-stu-id="04e4a-128">The changes are effective only when the application starts or restarts.</span></span> <span data-ttu-id="04e4a-129">Her iki öznitelik ayarlandığında bir olay başlangıçta kaydedilir `true`.</span><span class="sxs-lookup"><span data-stu-id="04e4a-129">An event is logged at startup when both attributes are set to `true`.</span></span> <span data-ttu-id="04e4a-130">Bir olay, ayrıca kaydedilir `logKnownPii` ayarlanır `true` ancak `enableLoggingKnownPii` olan `false`.</span><span class="sxs-lookup"><span data-stu-id="04e4a-130">An event is also logged if `logKnownPii` is set to `true` but `enableLoggingKnownPii` is `false`.</span></span>  
  
 <span data-ttu-id="04e4a-131">Makine Yöneticisi ve bu iki anahtar kullanıldığında, uygulama dağıtıcı son derece dikkatli olun çalışma.</span><span class="sxs-lookup"><span data-stu-id="04e4a-131">The machine administrator and application deployer should exercise extreme caution when using these two switches.</span></span> <span data-ttu-id="04e4a-132">PII günlüğe kaydetme işlemi etkinleştirilmişse, güvenlik anahtarlarını ve PII günlüğe kaydedilir.</span><span class="sxs-lookup"><span data-stu-id="04e4a-132">If PII logging is enabled, security keys and PII are logged.</span></span> <span data-ttu-id="04e4a-133">Devre dışı ise, hala ileti üstbilgilerini ve gövdeleri hassas ve uygulamaya özgü veriler günlüğe kaydedilir.</span><span class="sxs-lookup"><span data-stu-id="04e4a-133">If it is disabled, sensitive and application-specific data is still logged in message headers and bodies.</span></span> <span data-ttu-id="04e4a-134">Gizlilik ve maruz kalma PII koruma hakkında daha kapsamlı bir açıklama için bkz: [kullanıcı gizliliği](http://go.microsoft.com/fwlink/?LinkID=94647).</span><span class="sxs-lookup"><span data-stu-id="04e4a-134">For a more thorough discussion about privacy and protecting PII from being exposed, see [User Privacy](http://go.microsoft.com/fwlink/?LinkID=94647).</span></span>  
  
> [!CAUTION]
>  <span data-ttu-id="04e4a-135">PII hatalı biçimlendirilmiş iletilerinde gizli değildir.</span><span class="sxs-lookup"><span data-stu-id="04e4a-135">PII is not hidden in malformed messages.</span></span> <span data-ttu-id="04e4a-136">Olarak oturum gibi messaged-herhangi bir değişiklik yapılmadan olduğu.</span><span class="sxs-lookup"><span data-stu-id="04e4a-136">Such messaged are logged as-is without any modification.</span></span> <span data-ttu-id="04e4a-137">Belirtilen öznitelikler, daha önce bu üzerinde etkisi yoktur.</span><span class="sxs-lookup"><span data-stu-id="04e4a-137">Attributes mentioned previously have no effect on this.</span></span>  
  
### <a name="custom-trace-listener"></a><span data-ttu-id="04e4a-138">Özel İzleme dinleyicisi</span><span class="sxs-lookup"><span data-stu-id="04e4a-138">Custom Trace Listener</span></span>  
 <span data-ttu-id="04e4a-139">Özel İzleme dinleyicisi ileti oturum açma ekleme izleme yöneticinin kısıtlanmış bir ayrıcalık kaynağıdır.</span><span class="sxs-lookup"><span data-stu-id="04e4a-139">Adding a custom trace listener on the Message Logging trace source is a privilege that should be restricted to the administrator.</span></span> <span data-ttu-id="04e4a-140">Kötü amaçlı özel dinleyicileri uzaktan iletileri göndermek için yapılandırılmış olmasıdır hassas bilgilerin gösterilmesine yol açar.</span><span class="sxs-lookup"><span data-stu-id="04e4a-140">This is because malicious custom listeners can be configured to send messages remotely, which leads to sensitive information disclosure.</span></span> <span data-ttu-id="04e4a-141">Ayrıca, kablo, iletileri gibi uzak bir veritabanına göndermek için özel bir dinleyici yapılandırırsanız, uzak makinede ileti günlüklerini uygun erişim denetimini uygulamalıdır.</span><span class="sxs-lookup"><span data-stu-id="04e4a-141">In addition, if you configure a custom listener to send messages on the wire, such as, to a remote database, you should enforce proper access control on the message logs in the remote machine.</span></span>
